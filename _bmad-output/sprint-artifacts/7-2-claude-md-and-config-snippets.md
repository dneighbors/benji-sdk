## Story

As a developer,
I want a CLAUDE.md documenting repo conventions for AI assistants,
so that Claude Code can work effectively in this repo.

## Prerequisites

- **Epic 1** -- Monorepo + Core Infrastructure (`done`): Monorepo structure, packages, shared auth/errors all in place.
- **Story 7.1** -- README and MCP usage docs (`done`): Root README.md already exists with project overview, MCP config snippets, CLI docs, env vars. CLAUDE.md must complement (not duplicate) the README.

## Background

The project is a pnpm monorepo with three packages (`benji-sdk`, `benji-mcp`, `benji-cli`). Story 7.1 created a comprehensive root `README.md` covering setup, MCP config snippets, CLI usage, env vars, and the full MCP tool reference. What is still missing is a `CLAUDE.md` file at the project root -- the file that Claude Code reads automatically when working in a repository.

A good CLAUDE.md serves a fundamentally different purpose from a README. The README is for human developers learning about the project. CLAUDE.md is an instruction file for AI coding assistants, telling them:
- What commands to run (build, test, lint)
- What files are auto-generated and must not be hand-edited
- What patterns to follow when adding new code
- What conventions exist (naming, file placement, import style)
- What pitfalls to avoid

No `CLAUDE.md` currently exists in the repository. No `.claude/` directory commands file exists either.

### Task Group F: Authoring patterns (AC: #4, #5)

- [x] **F.1** Write the **Adding a New MCP Tool Domain** section with numbered step-by-step instructions:
  1. Create `packages/benji-mcp/src/tools/<domain>.ts`
  2. Import: `McpServer` from `@modelcontextprotocol/sdk/server/mcp.js`, `z` from `zod`, SDK resource class and `wrapSdkCall` from `benji-sdk`, `toolResult` and `handleToolError` from `./util.js`
  3. Export `function register<Domain>Tools(server: McpServer): void`
  4. For each tool, call `server.registerTool("<action>_<resource>", { description, inputSchema: { ... Zod schemas } }, async handler)`
  5. In the handler: `try { const result = await wrapSdkCall(SdkClass.method({ body: { ...params } })); return toolResult(result); } catch (error) { return handleToolError(error); }`
  6. Tool naming convention: `<action>_<resource>` in snake_case (e.g., `list_tags`, `create_todo`, `delete_weight_log`)
  7. Register in `packages/benji-mcp/src/server.ts`: import the function and call it with the `mcpServer` instance
  8. For date inputs, use the shared Zod schemas: `ymdDateSchema` for query/list operations, `tzDateSchema` for create/update operations (both from `./util.js`)
  (AC: #4)

- [x] **F.2** Write the **Adding a New CLI Command Domain** section with numbered step-by-step instructions:
  1. Create `packages/benji-cli/src/commands/<domain>.ts`
  2. Import: `Command` from `commander`, SDK resource class and `wrapSdkCall` from `benji-sdk`, `ensureAuth` from `../auth.js`, `getGlobalOptions` and `outputResult` from `../output.js`, `handleCommandError` from `../error-handler.js`, shared helpers as needed from `./shared.js`
  3. Export `function register<Domain>Command(program: Command): void`
  4. Create a top-level command group: `const cmd = program.command("<domain>").description("Manage <domain>")`
  5. Add subcommands: `cmd.command("<action>").description("...").option("--flag <value>", "...").action(async (args, options, command) => { ... })`
  6. In each action: call `ensureAuth()`, then `const opts = getGlobalOptions(command)`, then `try { const result = await wrapSdkCall(...); outputResult(result, opts); } catch (error) { handleCommandError(error); }`
  7. For delete commands, add `--force` option and call `requireForce(command, "<domain>", "delete")` before `ensureAuth()`
  8. For boolean fields, use `--flag` / `--no-flag` pairs and check `command.getOptionValueSource("flag") === "cli"` before including in the request body
  9. Support `--stdin` option for JSON input via `readStdin()`
  10. Register in `packages/benji-cli/src/commands/index.ts`: import and call the register function. The `addGlobalOptionsToLeaves` call at the end of `registerCommands` automatically adds `--json` and `--compact` to all leaf commands.
  (AC: #5)

### Naming conventions observed in codebase

- **MCP tool files**: `packages/benji-mcp/src/tools/<domain>.ts` -- kebab-case filenames (e.g., `pain-events.ts`, `weight-logs.ts`, `todo-lists.ts`)
- **MCP register functions**: `register<PascalDomain>Tools(server)` (e.g., `registerTagTools`, `registerPainEventTools`, `registerWeightLogTools`)
- **MCP tool names**: `<action>_<resource>` snake_case (e.g., `list_tags`, `create_todo`, `delete_weight_log`, `get_hydration_stats`)
- **CLI command files**: `packages/benji-cli/src/commands/<domain>.ts` -- kebab-case filenames matching MCP
- **CLI register functions**: `register<PascalDomain>Command(program)` (e.g., `registerTagsCommand`, `registerPainEventsCommand`)
- **CLI command structure**: `benji <resource> <action> [options]` (e.g., `benji tags list`, `benji todos create`)
- **SDK class methods**: `ClassName.classNameMethod()` camelCase (e.g., `Tags.tagsList()`, `Todos.todosCreate()`, `Habits.habitsGetHabitsAndCompletions()`)
